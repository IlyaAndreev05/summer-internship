# Задача 2. Разработка ИИ-консультанта по работе с ALINA GPSS

**Цель:** разработать ИИ-консультанта на основе LLM, который отвечает на вопросы пользователей по работе с ALINA GPSS на основе руководства пользователя, а после завершения Задачи 1 – также на основе библиотеки эталонных примеров. Работа состоит из двух частей: Часть 1 – локальное консольное приложение, Часть 2 – чат-бот на внешней платформе (например, ВК).

**В результате работы должно быть сформировано:**

### Часть 1:

- консольное приложение на основе LLM, отвечающее на вопросы по руководству пользователя ALINA GPSS;
- расширение приложения после подключения библиотеки эталонных примеров (результат Задачи 1): поиск и отображение релевантных примеров;
- тест-план и отчёт о тестировании, подтверждающий долю корректных ответов;
- инструкция администратора: установка, настройка, подключение источников данных и подключение новых репозиториев Git с обновлёнными данными (см. пример 3);
- инструкция пользователя по работе с приложением (см. пример 4);
- исходный код в репозитории с инструкцией по запуску.

### Часть 2:

- чат-бот с той же функциональностью на выбранной платформе (например, ВК);
- инструкция администратора по развёртыванию и подключению бота к платформе;
- инструкция пользователя по работе с ботом.

**Исходные материалы:** руководство пользователя ALINA GPSS; после завершения Задачи 1 – библиотека эталонных примеров в формате карточек проектов (см. Задача 1, пример 4).

**Технологический стек** выбирается самостоятельно при согласовании с преподавателем.

**Примечание по архитектуре:** система строится по принципу «поиск + генерация» (RAG): готовая языковая модель используется без дополнительного обучения/тонкой настройки – она генерирует ответ на основе фрагментов, найденных поисковым механизмом. Подготовка данных в этой схеме – это структурирование и разбиение текстовых источников (руководства, библиотеки примеров) для последующего поиска. Сам механизм индексации/поиска может быть реализован студентами вручную или предоставлен выбранным RAG-фреймворком/библиотекой – это часть решения, которое выбирается на Этапе 1.

**Принцип работы системы:**

1. Пользователь задаёт вопрос в свободной форме.
2. Система ищет релевантные фрагменты в проиндексированных источниках (руководство пользователя, описания проектов).
3. На основе найденных фрагментов языковая модель формирует ответ.
4. При наличии подходящего проекта – система предлагает его как пример.

## Этапы работы

### Этап 1: Подготовка и проектирование

Изучить руководство пользователя ALINA GPSS. Выбрать технологический стек и согласовать с преподавателем. Определить архитектуру системы: способ индексации документов, механизм поиска релевантных фрагментов, языковую модель, интерфейс. Подготовить план разработки с дедлайнами.

**Результатом работы этапа 1** является согласованная архитектура и план разработки.

### Этап 2: Разработка базового прототипа (консольное приложение)

Подготовить и структурировать руководство пользователя как источник знаний: разбить текст на смысловые фрагменты, при необходимости снабдить их метаданными (раздел, заголовок) (см. пример 1). Реализовать или подключить механизм поиска релевантных фрагментов по этим данным. Реализовать цепочку: вопрос пользователя → поиск релевантных фрагментов → генерация ответа. Развернуть базовый пользовательский интерфейс в виде консольного приложения. Провести первичное тестирование на 20+ вопросах по документации.

**Результатом работы этапа 2** является работающий прототип, отвечающий на вопросы по руководству пользователя.

### Этап 3: Подключение библиотеки эталонных примеров

Получить от Задачи 1 файл карточек библиотеки эталонных примеров (см. Задача 1, пример 4). Добавить описания проектов в поисковый индекс. Реализовать логику: при вопросах о примерах применения – предлагать релевантный проект из библиотеки. Обновить интерфейс: отображать найденные примеры отдельным блоком.

**Результатом работы этапа 3** является приложение, дополненное поиском по библиотеке эталонных примеров.

### Этап 4: Тестирование и документирование

Составить тест-план (см. пример 2): не менее 50 вопросов разных типов (справочные, практические, по примерам). Оценить качество ответов (доля корректных ответов — не менее 70%), выявить слабые места. Доработать систему по результатам тестирования. Написать инструкцию администратора (см. пример 3) и инструкцию пользователя (см. пример 4).

**Результатом работы этапа 4** является протестированное консольное приложение с полным комплектом документации.

### Этап 5: Реализация в виде чат-бота

На основе той же логики и того же поискового индекса перенести функциональность в формат чат-бота на выбранной платформе (например, ВК). Адаптировать интерфейс под формат диалога платформы. Повторно прогнать тест-план (см. пример 2) в новом интерфейсе. Написать инструкцию администратора по развёртыванию бота и подключению к платформе, а также инструкцию пользователя по работе с ботом.

**Результатом работы этапа 5** является чат-бот с той же функциональностью, развёрнутый на внешней платформе.

## Пример 2. Структура тест-плана

| № | Тип вопроса | Вопрос | Критерий корректного ответа | Ответ системы | Результат (корректно/некорректно) | Комментарий |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | справочный | Например: «Что такое транзакт?» | Краткое и точное определение по руководству | — | — | — |

Типы вопросов: справочные (по теории/терминам), практические (как сделать действие), по примерам (есть ли проект на похожую задачу).

## Пример 3. Структура инструкции администратора

1. Требования к окружению (ОС, зависимости, версии).
2. Установка и настройка (шаги, переменные окружения, доступы к LLM).
3. Подключение источников данных: руководство пользователя, библиотека эталонных примеров (путь/репозиторий Git).
4. Подключение нового репозитория Git с обновлёнными данными.
5. Запуск приложения.
6. Типичные ошибки и их решения.

---

## Пример 4. Структура инструкции пользователя

1. Как запустить приложение.
2. Как задать вопрос и какие типы вопросов поддерживаются.
3. Как интерпретировать ответ (в т.ч. блок с примером из библиотеки).
4. Известные ограничения системы.
