# Задача 1. Создание библиотеки эталонных проектов

**Цель:** необходимо провести аудит накопленных студенческих проектов по имитационному моделированию, присвоить каждому проекту градацию качества, отобрать лучшие проекты и подготовить из них библиотеку эталонных примеров.

## Результат работы

- сводная таблица всех проанализированных проектов с присвоенной градацией;
- библиотека из 50 (по 10 проектов на человека) доработанных и оформленных проектов;
- единая структура хранения материалов в репозитории Git (см. Пример 1).

**Исходные материалы:** для выполнения задания предоставляется набор ранее выполненных проектов по имитационному моделированию. Каждый проект включает: файл модели; отчет; материалы по анимации и визуализации (при наличии).

---

## Этапы работы

### Этап 1: Анализ предоставленных проектов

На первом этапе необходимо: собрать все предоставленные проекты в единый каталог (**архив/**); изучить содержание каждого проекта; определить предметную область; оценить: корректность модели, качество отчета и оформления (см. Пример 6); присвоить проекту градацию; зафиксировать результат в сводном файле (см. Пример 2).

Оценка проекта проводится по трём независимым направлениям:

- **Статус модели:** корректна; требует доработки (есть ошибки в логике/расчётах, но исправимо); неработоспособна (критические ошибки, нет смысла восстанавливать).
- **Статус отчёта:** готов; требует доработки (неполный/неструктурированный/слабые выводы); отсутствует или не подлежит восстановлению.
- **Статус оформления:** готово; требует доработки.

На основе этих трёх статусов итоговая градация проекта определяется автоматически:

- **Непригодно** – если статус модели = «Неработоспособна» ИЛИ статус отчёта = «Отсутствует/не подлежит восстановлению».
- **Готово** – если все три статуса (Корректна / Готов / Готово).
- **Требует доработки** – во всех остальных случаях, при этом указывается, по какому именно направлению(ям): модель / отчёт / оформление.

**Результатом работы этапа 1** является полный список проектов с указанием их градации и комментария по выявленным недостаткам.

### Этап 2: Отбор 50 проектов для включения в библиотеку примеров

После аудита необходимо отобрать 50 проектов (по 10 проектов на человека), которые будут включены в библиотеку эталонных примеров.

**Результатом работы этапа 2** является отбор проектов. Проекты должны покрывать различные предметные области, а также различные сочетания направлений доработки (модель / отчёт / оформление). Проекты с итоговой градацией «Непригодно» не рассматриваются.

### Этап 3: Приведение отобранных проектов к единому стандарту

Для каждого отобранного проекта необходимо: проверить работоспособность модели; при необходимости исправить ошибки в модели; сверить отчет с эталонной структурой (см. Пример 3); доработать отчет до единого стандарта; при необходимости заново выполнить эксперименты, если результаты исходного отчета некорректны или модель была изменена; при наличии анимации структурной схемы перенести соответствующие материалы; при отсутствии анимации и наличии времени разработать ее дополнительно.

**Результатом работы этапа 3** является набор проектов, приведенных к единому стандарту: исправленные и проверенные файлы моделей; доработанные отчеты по эталонной структуре; обновленные результаты экспериментов и материалы по анимации структурной схемы (при наличии).

### Этап 4: Формирование библиотеки эталонных примеров

Для каждого отобранного и скорректированного проекта необходимо подготовить папку, в которую поместить файл с рабочей моделью и отчет в установленной структуре (см. Пример 3). Подготовить единый файл карточек отобранных проектов (см. Пример 4).

**Результатом работы этапа 4** является библиотека эталонных примеров, содержащая 50 полностью оформленных проектов, пригодных для демонстрации и дальнейшего использования в учебной и практической деятельности. А также единый файл карточек проектов для поиска и фильтрации.

## Пример 1. Требуемая структура хранения в Git

В одном репозитории используются две верхнеуровневые папки:

- **архив/** – исходные проекты;
- **библиотека примеров/** – 50 отобранных и доработанных проектов.

Для навигации и учета используется сводный файл:

- **в архив/** – файл с результатами аудита и градацией всех проектов (см. Пример 2);
- **в библиотека примеров/** – единый файл карточек проектов для поиска и фильтрации по библиотеке (см. Пример 4).

Проекты, добавленные в **библиотеку примеров/**, удаляются из **архива/**.

## Пример 3. Требуемая структура отчёта по каждому проекту

### Глава 1. Разработка имитационной модели

#### 1.1 Описание системы

Краткое описание объекта моделирования, целей исследования и исходных данных.

#### 1.2 Формализация системы

Описание моделируемой системы, включая:

- перечень объектов системы (транзакты, устройства, очереди, накопители);
- описание взаимодействий между объектами;
- ограничения и допущения модели;
- схема процесса (блок-схема или концептуальная диаграмма).

#### 1.3 Разработка имитационной модели

Реализация модели в ALINA GPSS, включая:

- структурную схему модели (скриншот);
- описание каждого функционального блока: назначение, логика работы, скриншот блока.

### Глава 2. Конструирование имитационного приложения

#### 2.1 Форма «Ввод данных»

Описание формы ввода исходных данных: какие параметры задаются, как организованы группы параметров, скриншот формы.

#### 2.2 Форма «Планирование экспериментов»

Описание формы планирования: выбранные факторы и их диапазоны, целевые показатели, скриншот формы.

#### 2.3 Форма «Ролик динамики»

Описание формы динамики хода эксперимента: какие показатели отображаются, вкладки и графики, скриншоты.

### Глава 3. Имитационное исследование

Для каждого эксперимента необходимо указать: цель, исходные данные запуска, полученные результаты (скриншоты), анализ результатов, выводы.

#### 3.1 Одиночные эксперименты

Один или несколько одиночных прогонов модели с фиксированными параметрами. Первичная оценка поведения системы.

#### 3.2 Серии экспериментов

Последовательные запуски при варьировании одного или нескольких факторов. Анализ влияния параметров на выходные показатели.

#### 3.3 Оптимизационные эксперименты

Поиск рациональных значений управляемых параметров. Описание критерия оптимизации и полученных результатов.

### Глава 4. Анимация структурной схемы (при наличии)

#### 4.1 Элементы управления

Описание элементов управления, их настройки и примеры использования (скриншоты).

#### 4.2 Элементы визуализации

Описание элементов визуализации, их настройки и примеры отображения результатов (скриншоты).

## Пример 6. Критерии оценки

**Корректность модели:** модель воспроизводима (повторный запуск даёт стабильные, ожидаемые результаты); логика модели не содержит явных ошибок (соответствие реальному процессу, отсутствие зависаний/тупиков, корректная обработка транзактов); исходные параметры и полученные результаты согласованы между собой и с описанием в отчёте; эксперименты выполнены с достаточным числом прогонов/времени моделирования для значимости результатов (где это применимо).

**Качество отчёта:** присутствуют все обязательные разделы по эталонной структуре (см. Пример 3); результаты экспериментов подтверждены данными (таблицы, графики, скриншоты), а не только описанием; выводы обоснованы и соответствуют полученным результатам; нет логических разрывов между разделами (например, цель эксперимента соответствует представленным результатам).

**Качество оформления:** единообразное оформление таблиц, рисунков, подписей по всему отчёту; структура читаема (заголовки, нумерация разделов, отсутствие «сырых» фрагментов текста); скриншоты читаемы и подписаны; нет опечаток/ошибок, мешающих восприятию.

# Задача 2. Разработка ИИ-консультанта по работе с ALINA GPSS

**Цель:** разработать ИИ-консультанта на основе LLM, которое отвечает на вопросы пользователей по работе с ALINA GPSS на основе руководства пользователя, а после завершения Задачи 1 – также на основе библиотеки эталонных примеров. Работа состоит из двух частей: Часть 1 – локальное консольное приложение, Часть 2 – чат-бот на внешней платформе (например, ВК).

## Результат работы

### Часть 1

- консольное приложение на основе LLM, отвечающее на вопросы по руководству пользователя ALINA GPSS;
- расширение приложения после подключения библиотеки эталонных примеров (результат Задачи 1): поиск и отображение релевантных примеров;
- тест-план и отчёт о тестировании, подтверждающий долю корректных ответов;
- инструкция администратора: установка, настройка, подключение источников данных и подключение новых репозиториев Git с обновлёнными данными (см. Пример 3);
- инструкция пользователя по работе с приложением (см. Пример 4);
- исходный код в репозитории с инструкцией по запуску.

### Часть 2

- чат-бот с той же функциональностью на выбранной платформе (например, ВК);
- инструкция администратора по развёртыванию и подключению бота к платформе;
- инструкция пользователя по работе с ботом.

**Исходные материалы:** руководство пользователя ALINA GPSS; после завершения Задачи 1 – библиотека эталонных примеров в формате карточек проектов (см. Задача 1, Пример 4).

**Технологический стек** выбирается самостоятельно при согласовании с преподавателем.

**Примечание по архитектуре:** система строится по принципу «поиск + генерация» (RAG): готовая языковая модель используется без дополнительного обучения/тонкой настройки – она генерирует ответ на основе фрагментов, найденных поисковым механизмом. Подготовка данных в этой схеме – это структурирование и разбиение текстовых источников (руководства, библиотеки примеров) для последующего поиска. Сам механизм индексации/поиска может быть реализован студентами вручную или предоставлен выбранным RAG-фреймворком/библиотекой – это часть решения, которое выбирается на Этапе 1.

### Принцип работы системы

1. Пользователь задаёт вопрос в свободной форме.
2. Система ищет релевантные фрагменты в проиндексированных источниках (руководство пользователя, описания проектов).
3. На основе найденных фрагментов языковая модель формирует ответ.
4. При наличии подходящего проекта – система предлагает его как пример.

---

## Этапы работы

### Этап 1: Подготовка и проектирование

Изучить руководство пользователя ALINA GPSS. Выбрать технологический стек и согласовать с преподавателем. Определить архитектуру системы: способ индексации документов, механизм поиска релевантных фрагментов, языковую модель, интерфейс. Подготовить план разработки с дедлайнами.

**Результатом работы этапа 1** является согласованная архитектура и план разработки.

### Этап 2: Разработка базового прототипа (консольное приложение)

Подготовить и структурировать руководство пользователя как источник знаний: разбить текст на смысловые фрагменты, при необходимости снабдить их метаданными (раздел, заголовок) (см. Пример 1). Реализовать или подключить механизм поиска релевантных фрагментов по этим данным. Реализовать цепочку: вопрос пользователя → поиск релевантных фрагментов → генерация ответа. Развернуть базовый пользовательский интерфейс в виде консольного приложения. Провести первичное тестирование на 20+ вопросах по документации.

**Результатом работы этапа 2** является работающий прототип, отвечающий на вопросы по руководству пользователя.

### Этап 3: Подключение библиотеки эталонных примеров

Получить от Задачи 1 файл карточек библиотеки эталонных примеров (см. Задача 1, Пример 4). Добавить описания проектов в поисковый индекс. Реализовать логику: при вопросах о примерах применения – предлагать релевантный проект из библиотеки. Обновить интерфейс: отображать найденные примеры отдельным блоком.

**Результатом работы этапа 3** является приложение, дополненное поиском по библиотеке эталонных примеров.

### Этап 4: Тестирование и документирование

Составить тест-план (см. Пример 2): не менее 50 вопросов разных типов (справочные, практические, по примерам). Оценить качество ответов (доля корректных ответов — не менее 70%), выявить слабые места. Доработать систему по результатам тестирования. Написать инструкцию администратора (см. Пример 3) и инструкцию пользователя (см. Пример 4).

**Результатом работы этапа 4** является протестированное консольное приложение с полным комплектом документации.

### Этап 5: Реализация в виде чат-бота

На основе той же логики и того же поискового индекса перенести функциональность в формат чат-бота на выбранной платформе (например, ВК). Адаптировать интерфейс под формат диалога платформы. Повторно прогнать тест-план (см. Пример 2) в новом интерфейсе. Написать инструкцию администратора по развёртыванию бота и подключению к платформе, а также инструкцию пользователя по работе с ботом.

**Результатом работы этапа 5** является чат-бот с той же функциональностью, развёрнутый на внешней платформе.

## Пример 3. Структура инструкции администратора

1. Требования к окружению (ОС, зависимости, версии).
2. Установка и настройка (шаги, переменные окружения, доступы к LLM).
3. Подключение источников данных: руководство пользователя, библиотека эталонных примеров (путь/репозиторий Git).
4. Подключение нового репозитория Git с обновлёнными данными.
5. Запуск приложения.
6. Типичные ошибки и их решения.

---

## Пример 4. Структура инструкции пользователя

1. Как запустить приложение.
2. Как задать вопрос и какие типы вопросов поддерживаются.
3. Как интерпретировать ответ (в т.ч. блок с примером из библиотеки).
4. Известные ограничения системы.
